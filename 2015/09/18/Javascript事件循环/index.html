从一段代码开始

Javascript API文档明确定义：setTimeout的第二个参数意义为隔多少ms后，回调方法就会被执行。那么可以推断出，这里设置成0ms，应该被立刻执行了。既然立即执行，为什么这段代码的作者要舍近求远呢？

显然并不是！！！显然并不是！！！显然并不是！！！

在javascript权威指南中有阐述：当setTimeout的延迟时间设置为0的时候，回调函数不会马上执行，而是进入 事件队列。

Javascript引擎是单线程模式运行的，浏览器无论在什么时候都只且只有一个javascript线程在运行程序。

浏览器事件处理

浏览器内核实现允许多个线程异步执行，这些线程在内核控制下相互配合以保持同步。假设某一浏览器内核的实现至少有3个常驻线程：javascript引擎线程、界面渲染线程、浏览器事件触发线程。除此之外，也有一些执行完就终止的线程，如HTTP请求线程等，这些异步线程都会产生不同的异步事件。

浏览器中的javascript引擎是基于事件驱动的，这里的事件可以看作是浏览器派给他的任务，如使用setTimeout添加一个任务，也可以来自浏览器内核的其他线程，如界面元素鼠标点击事件、定时触发器时间到达通知、异步请求状态变更通知等。

从代码的角度看任务实体，就是各种回调函数。javascript引擎一直等待着任务队列中任务的到来。由于单线程关系，这些任务需要排队一个接着一个被引擎处理。

GUI渲染线程

该线程负责渲染浏览器界面HTML元素，当界面需要重绘repaint或者由于某种操作引发回流reflow时，该线程就会执行。这个线程和javascript引擎线程是互斥的，这也很容易理解：javascript脚本是可以操控dom元素的，在修改元素的同时渲染界面，渲染线程前后获得的元素数据就可能不一致了。在javascript引擎运行脚本期间，浏览器渲染线程都是出于挂起状态的，也就是说被冻结了。所以，在脚本中执行对界面进行的更新操作，如添加节点、删除节点或者改变节点的外观等更新并不会立即体现出来，这些操作将保存在一个队列中，等待javascript引擎空闲时才有机会渲染出来。

JS事件触发线程

javascript脚本的执行不影响html元素事件的触发，在Time1时段内，用户点击鼠标键，点击事件被浏览器事件触发线程捕捉后形成一个鼠标点击事件，对于javascript引擎线程来说，这个事件是由其他线程异步传到任务队列队尾的，由于引擎正在处理Time1时段的任务，这个鼠标点击事件就会排队等待处理。

定时触发线程

这里的浏览器模型定时计数器并不是由javascript引擎计数的，因为javascript引擎是单线程的，如果处于阻塞线程状态就不能计时，它必须依赖外部计时来触发定时，所以队列中的定时事件也是异步事件。

setTimeout的坑坑

第一段代码进入了死循环，但不久就会发现，第二、第三行并不是预料中的事情，2个alert问候并未出现！！！

这个误区出现的最主要的原因是：潜意识中认为，javascript引擎有多个线程在执行，javascript的定时器回调函数是异步执行的。

正如上面介绍的浏览器事件处理流程那样，t1时段内，继鼠标点击事件触发后，先前已设置的setTimeout定时也到达了，此刻对javascript引擎来说，定时触发线程产生了一个异步定时事件被放到任务队列中，该事件被排到点击事件回调之后等待处理。同理，t1时段内，接下来某个setInterval定时器也被添加了，由于是间隔定时，在t1时段内连续被触发了2次，这些事件都被排到队尾等待处理。可见，假如t1非常长，远大于setInterval的定时间隔，那么定时触发线程就会源源不断地产生异步定时事件并放到任务队列尾部而不管他们是否已经被处理，但是一旦t1和最先的定时事件前面的任务已经处理完，这些排列中的定时事件就依次不间断的被执行，这是因为，对于javascript引擎而言，在处理队列中的各任务处理方式都是一样的，只是处理的次序不同而已。

  1. 案例

    setTimeout(function(){
    /* 代码块 */
    setTimeout(arguments.callee, 10);
}, 10);
setInterval(function(){
    /* 代码块 */
}, 10);

    这两段代码看起来效果一样，其实非也。第一段回调函数内的setTimeout是javascript引擎执行后再设置新的setTimeout定时，假定上一个回调函数处理完到下一个回调开始处理为一个时间间隔，理论上2个setTimeout回调执行时间间隔>=10ms。而第二段自setInterval设置定时后，定时触发线程就会源源不断地每隔10ms产生异步定时事件并放到任务队列尾部，理论上2个setInterval回调执行时间间隔<=10ms。

  2. ajax异步请求是否真的异步

    XMLHTTPRequest异步请求是由浏览器新开一个线程请求，当请求的状态变更时，如果先前已经设置回调，那么异步线程就产生状态变更事件放到javascript引擎的处理队列中等待处理。所以，当任务处理时，javascript引擎始终是单线程运行回调函数。

  3. 倒计时实现

    KISSY.add('timer', function(S) {
    var fns = [],
        commands = [];
    var d;    
    function timer() {
        while (commands.length) {
            commands.shift()();
        }
        var diff = +new Date() - timer.nextTime,
            count = 1 + Math.floor(diff / 100);
        diff = 100 - diff % 100;
        timer.nextTime += 100 * count;
        var frequency, step,
            i, len;
        for (i = 0, len = fns.length; i < len; i += 2) {
            frequency = fns[i + 1];
            if (0 === frequency) {
                count = count || 0;
                d -= 100 * count;
                fns[i](d);
            } else {
                frequency += 2 * count - 1;
                step = Math.floor(frequency / 20);
                if (step > 0) {
                    step = step || 0;
                    d -= 1000 * step;
                    fns[i](d);
                }
                fns[i + 1] = frequency % 20 + 1;
            }
        }
        setTimeout(timer, diff);
    }
    timer.nextTime = +new Date();    
    return {
        //获取服务器时间
        getServiceTime: function(callback) {
            var xmlhttp = new XMLHttpRequest;
            xmlhttp.onreadystatechange = function() {
                if (xmlhttp.readyState != 4) return;
                var now;
                if (!!xmlhttp.getResponseHeader('date')) {
                    now = new Date(xmlhttp.getResponseHeader('date'));
                }
                if (!now) now = new Date();
                xmlhttp.onreadystatechange = null;
                callback && callback(now);
            };
            xmlhttp.open("HEAD", location.href, true);
            xmlhttp.send(null);
        },
        add: function(fn, frequency) {
            commands.push(function() {
                fns.push(fn);
                fns.push(frequency === 1000 ? 1 : 0);
            });
        },
        remove: function(fn) {
            commands.push(function() {
                var i = S.indexOf(fn, fns);
                if (i !== -1) {
                    fns.splice(S.indexOf(fn, fns), 2);
                }
            });
        },
        start: function(){
            timer();
        },
        tick: function(endDate, frequency, callback) {
            var self = this;
            self.getServiceTime(function(date) {
                d = endDate.getTime() - date.getTime();
                callback && self.add(callback, frequency);
            });
        }
    };
});

非阻塞I/O

在javascript中，几乎所有的I/O都是非阻塞的。这包括HTTP请求、数据库操作和磁盘读写，单线程执行要求在运行期执行一个操作时，提供一个回调函数，然后继续做其他的事情。当操作依据完成时，消息和提供的回调函数一起插入到队列。在将来的某个时刻，消息从队列移除，回调函数被执行。

事件循环

将调用者和响应解耦，使用javascript在运行期等待异步操作完成和回调触发时可以做其他事情，但是这些回调在内存中是如何组织的，按什么顺序执行的呢？是什么导致他们被调用？

javascript运行时包含一个消息队列，它存储了需要处理的消息的列表和相关的回调函数。这些消息是以队列的形式来响应回调函数所涉及的外部事件（如鼠标单击或者收到HTTP请求的响应）。

在一次循环，队列提取下一条消息（每次提取称为一次tick）。当事件发生，该消息的回调执行。回调函数的调用在调用栈作为初始化片段，由于javascript是单线程的，未来的消息提取和处理是因为等待栈的所有调用返回而被停止。

同步函数调用

后续（同步）函数调用会添加新的调用frame到栈（例如，函数init调用函数changeColor）。

在这个例子中，当用户单击foo元素时，一条消息（及其回调函数changeColor）被插入到队列，并触发onclick事件。当消息离开队列时，其回调函数changeColor被调用，函数执行完毕时，事件循环仍在继续。只要函数changeColor存在，并指定为foo元素onclick方法的回调，那么在该元素的单击会导致更多的消息被插入到队列中去。

异步函数调用

如果一个函数在代码中按异步调用（比如setTimeout），提供的回调将最终作为一个不同的消息队列的一部分被执行，他将发生在事件循环的某个未来的动作上。

由于setTimeout的非阻塞特性，他的回调函数在至少0ms后触发，而不是作为消息的一部分被处理。在这个示例中，setTimeout被调用，传入了一个回调函数g且延迟0ms后执行。当我们指定事件到达后，一个单独的消息将被加入队列（g作为回调函数）。因此，控制台会打印foo baz blix bar。如果在同一个调用片段，两个调用都设置了setTimeout，且传递的第二个参数也相同，则它们的回调将按照调用顺序插入队列。

Web Worker

使用web worker允许您能够将一项费时的操作在一个单独的线程中执行，从而可以释放主线程去做别的事情。worker包括一个独立的消息队列、事件循环，内存空间独立于实例化他的原始线程。worker和主线程之间通过消息通信。

闭包

关于闭包这个小东西，我觉得需要单开一篇，现在只是简单介绍下闭包作为回调函数的一些小东东。

javascript对于闭包的支持，允许你这样注册回调函数，当回调函数执行时，保持了对它们被创建的环境的访问，即使回调的执行时创建了一个全新的调用栈。

在这个例子中，changeHeaderDeferred函数被执行时包含了变量header。函数setTimeout被调用，导致消息（带上changeHeader回调）被添加到消息队列，大约100ms后执行。然后changeHeaderDeferred函数返回false，结束第一个消息的处理，但header变量仍然可以通过闭包被引用，而不是被垃圾回收。当第二个消息被处理时，也就是changeHeader函数执行时，它保持了对在外部函数作用域中生命的header变量的访问。一旦其执行完毕，header变量将被垃圾回收。

结语

javascript事件驱动的交互模型不同于许多程序媛习惯的请求-响应模型，使用简单的消息队列和事件循环，使得开发人员在构建系统时使用大量的异步-触发回调函数，让运行时环境能够在等待外部事件触发的同时处理并发操作。
